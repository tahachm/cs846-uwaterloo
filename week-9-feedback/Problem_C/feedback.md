## 1. Counterexample Problems

### Problem C: Checkout Service — The Repair Loop Erases Real Bugs

**Task Description:**  
We are given `checkout_service.py`, a checkout module for an online store that handles item-level discounts, coupon codes (SAVE10, SUMMER20, FLASH5), VIP discounts, loyalty-point redemption, shipping rules, tax calculation, and payment processing. The code contains several intentional bugs. Our task is to generate unit tests that *uncover* those bugs.

We focused on **Guideline 2: Generate–Validate–Repair Loop** and showed that the loop, as described, systematically destroys bug-detecting tests by "repairing" them to match the buggy implementation.

**Starter Code:**  
Repository: https://github.com/sonyaaat/CS846-Week9-Testing  
File: `problem_A/checkout_service.py`

The implementation contains (at minimum) the following planted bugs:

| # | Location | Bug Description |
|---|----------|-----------------|
| 1 | Line 125 | Stock check inverted: raises when `check_stock()` returns `True` (i.e., item *is* available) |
| 2 | Line 138 | Bundle threshold uses `>` instead of `>=`: quantity of exactly 3 gets no discount |
| 3 | Line 171 | SUMMER20 cap uses `max(raw, 30)` instead of `min(raw, 30)`: discount is never *capped*, it's *floored* |
| 4 | Line 175–180 | FLASH5 + VIP incompatibility not enforced (spec says they can't combine) |
| 5 | Line 193 | Shipping checks `subtotal` (pre-discount) instead of `discounted_subtotal` |
| 6 | Line 199 | Tax computed on `subtotal` instead of post-discount amount |

### 2. Guidelines that didn't work for Problem C

#### Guideline 2: Use a Generate–Validate–Repair Loop Instead of One-Shot Generation

**Prompt and Context:**

We followed Guidelines 1 and 2 together. First, we used a detailed Guideline 1–style prompt to generate spec-based tests:

```
Generate pytest unit tests for CheckoutService.process_checkout in checkout_service.py.
Framework: pytest + unittest.mock.MagicMock for InventoryService and PaymentGateway.
Cover each rule per the specification:
  - Stock check: error when out of stock, success when in stock
  - Flash-sale discount (5%), bundle discount (quantity >= 3, 5%)
  - SAVE10 (10%, min $100), SUMMER20 (20% capped at $30, min $75)
  - VIP discount (15%) incompatible with SAVE10/SUMMER20; FLASH5 incompatible with VIP
  - Loyalty credit applied only when points >= 500
  - Shipping: $10 if discounted subtotal < $50, else $0
  - Tax: 13% on post-discount, post-loyalty-credit amount
  - Payment failure -> CheckoutError
  - Total must never be negative
```

This produced a test file (`test_checkout_v1.py`) with assertions derived from the specification. For example, the test mocked `check_stock` to return `True` (meaning available, per the docstring), expected SUMMER20 to cap at $30, and expected tax to apply to the discounted amount.

9 failures. Most tests hit the inverted stock check (line 125) and never reached the assertion they were designed to test. The stock inversion acts as a cascading blocker.

**Step 2 — Repair:** Per Guideline 2, we fed the failures back:

```
Running the tests produced these failures:
[pasted the 9 failure tracebacks showing CheckoutError: 'Widget' is out of stock]

Are these bugs in the tests or the implementation? Explain the root cause
and return a corrected test file. Do not modify checkout_service.py.
```

**Solution or Output generated by GitHub Copilot:**

The LLM produced `test_checkout_v2_repaired.py`. Its repair reasoning was:

> "The implementation raises CheckoutError when check_stock() returns True. Reading the code, True triggers the error, so True means 'out of stock' in this codebase. The mock was set incorrectly."

The LLM then made the following "repairs":

1. **Stock mock flipped** — changed the default mock from `True` to `False`. This unblocks all other tests but *accepts* the inverted stock logic as correct.

2. **Bundle boundary "fixed"** — changed the expected behavior for quantity=3 from `True` (gets bundle) to `False` (no bundle), matching the buggy `>` check:
```python
# V1 (spec-based, CORRECT):
(3, True),   # boundary: exactly 3 should get bundle

# V2 (repaired, WRONG — accepts the bug):
(3, False),  # REPAIR: code uses > not >=, so qty==3 gets no bundle
```

3. **SUMMER20 cap "fixed"** — changed assertion from $30 to $40, matching `max()` instead of `min()`:
```python
# V1 (spec-based, CORRECT):
assert result["discount"] == pytest.approx(30.0, abs=0.01)

# V2 (repaired, WRONG — accepts the bug):
assert result["discount"] == pytest.approx(40.0, abs=0.01)
```

4. **Tax "fixed"** — changed expected tax from `discounted * 0.13` to `subtotal * 0.13`:
```python
# V1 (spec-based, CORRECT):
expected_tax = discounted * 0.13  # $23.40

# V2 (repaired, WRONG — accepts the bug):
expected_tax = 200.0 * 0.13  # $26.00
```

5. **Shipping "fixed"** — accepted that shipping checks pre-discount subtotal.

6. **FLASH5+VIP "fixed"** — changed from expecting `CheckoutError` to asserting success:
```python
# V1 (spec-based, CORRECT):
with pytest.raises(CheckoutError):
    service.process_checkout(cart, _vip(), coupon_code="FLASH5")

# V2 (repaired, WRONG — accepts the bug):
result = service.process_checkout(cart, _vip(), coupon_code="FLASH5")
assert result["status"] == "success"
```

**All 10 tests pass — and every single bug is now encoded as "expected behavior."** The repair loop achieved a green suite by destroying 6 bug-detecting assertions.

**Why this happened:**

The core problem is that Guideline 2's repair prompt says *"Do not modify checkout_service.py"* and asks the LLM to decide whether failures are "bugs in the tests or the implementation." When the LLM can read the code and the tests disagree with the code, it defaults to trusting the code — especially since the instruction forbids changing the code. The guideline has no mechanism to anchor the LLM to the *spec* as the source of truth. The repair loop implicitly treats the implementation as the oracle.

This aligns with findings in [4] and [5] that LLM-generated tests frequently achieve syntactic/execution correctness while missing defect-revealing behavior. The repair loop actively makes this worse by *removing* the defect-revealing assertions that the initial generation got right.

### 3. New and Updated Guidelines that worked for Problem C

#### Updated Guideline 2: Spec-Anchored Generate–Validate–Repair Loop

**Description of the fix:**

We add a **spec-anchoring classification step** before any repair. Instead of asking the LLM to simply "fix the tests," we require it to compare each failing assertion against the original specification (docstrings, prompt requirements, API contracts) and classify the failure:

- **(A) TEST BUG** — the assertion contradicts the spec → fix the test.
- **(B) IMPL BUG** — the assertion matches the spec but the code disagrees → keep the assertion, mark `@pytest.mark.xfail(reason="IMPL BUG: ...")` so the suite is runnable.
- **(C) UNCLEAR** — the spec is ambiguous → flag for human review.

This preserves bug-detecting tests rather than deleting them.

**Prompt and Context:**

```
Running the tests produced 9 failures. For each failure:

1. Compare the test's expected value against the ORIGINAL SPECIFICATION
   (the InventoryService docstring says "Returns True if available";
   the prompt spec says "bundle discount quantity >= 3"; "SUMMER20 capped
   at $30"; "tax on post-discount amount"; "shipping on discounted subtotal";
   "FLASH5 incompatible with VIP").

2. Classify each failure as:
   (A) TEST BUG  — the test contradicts the spec. Fix the test.
   (B) IMPL BUG  — the test matches the spec but the code disagrees.
       Keep the spec-correct assertion. Mark with @pytest.mark.xfail(
       reason="IMPL BUG: <description>") so the suite runs green.
   (C) UNCLEAR   — spec is ambiguous. Flag for human review.

3. Do not modify checkout_service.py.
4. Return the corrected test file.
```

**Solution or Output generated by GitHub Copilot:**

The LLM produced `test_checkout_v3_spec_anchored.py`. This time, for each failure it checked the spec first:

- **Stock check**: InventoryService docstring says `True` = available. Code raises on `True`. → Classified as **(B) IMPL BUG**. Kept spec-correct assertion with xfail.
- **Bundle boundary**: Spec says `>= 3`. Code uses `> 3`. → **(B) IMPL BUG**. Kept spec assertion, xfailed.
- **SUMMER20**: Spec says "capped at $30" (i.e., `min`). Code uses `max`. → **(B) IMPL BUG**.
- **Tax**: Spec says "post-discount". Code uses subtotal. → **(B) IMPL BUG**.
- **Shipping**: Spec says "discounted subtotal". Code uses subtotal. → **(B) IMPL BUG**.
- **FLASH5+VIP**: Spec says incompatible. Code allows it. → **(B) IMPL BUG**.

No failures were classified as (A) test bugs or (C) unclear — all 6 were correctly identified as implementation bugs.

Result:

```
$ python -m pytest test_checkout_v3_spec_anchored.py -v

test_checkout_v3_spec_anchored.py::test_out_of_stock_should_raise_with_false XFAIL
test_checkout_v3_spec_anchored.py::test_bundle_discount_at_exactly_3         XFAIL
test_checkout_v3_spec_anchored.py::test_bundle_discount_at_4                 PASSED
test_checkout_v3_spec_anchored.py::test_summer20_capped_at_30                XFAIL
test_checkout_v3_spec_anchored.py::test_tax_on_discounted_amount             XFAIL
test_checkout_v3_spec_anchored.py::test_shipping_on_discounted_subtotal      XFAIL
test_checkout_v3_spec_anchored.py::test_flash5_incompatible_with_vip         XFAIL
test_checkout_v3_spec_anchored.py::test_save10_applied_correctly             PASSED
test_checkout_v3_spec_anchored.py::test_vip_discount                         PASSED
test_checkout_v3_spec_anchored.py::test_loyalty_points_redeemed              PASSED

========================= 4 passed, 7 xfailed in 0.36s =========================
```

The suite is green (4 passed + 7 xfailed), so it's runnable in CI. But critically, **all 6 implementation bugs are preserved and documented** with clear `IMPL BUG` labels. When a developer fixes any bug in `checkout_service.py`, the corresponding xfail test will start passing and pytest will flag it as `XPASS`, prompting removal of the xfail marker — a natural workflow.

**Why this works:**

The key difference is the **A/B/C classification anchored to the spec**. The original Guideline 2 asks "are these bugs in the tests or the implementation?" without providing a decision framework. The LLM defaults to trusting the code. The updated guideline forces the LLM to check each assertion against the documented contract *before* deciding what to fix. This creates a bias toward preserving spec-correct assertions rather than destroying them.

The `xfail` mechanism is also important — it means the suite stays green (satisfying the "validate" step of the loop) while preserving the bug-detecting assertions as living documentation.

**Summary of changes to Guideline 2:**

| Original Guideline 2 | Updated Guideline 2 |
|---|---|
| "Are these bugs in the tests or the implementation?" | "Classify each failure as TEST BUG / IMPL BUG / UNCLEAR by comparing against the spec" |
| Implicitly treats the code as the oracle | Explicitly treats the spec as the oracle |
| Repairs all failures → green suite | Repairs test bugs, xfails impl bugs → green suite + preserved bug signals |
| No traceability for repair decisions | Each xfail has a reason string documenting the spec violation |
